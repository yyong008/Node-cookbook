              验证之后，由GraphQL服务器执行GraphQL查询，该服务器返回的结果将反映所请求查询的形状，通常为JSON。

              GraphQL在没有类型系统的情况下无法执行查询，让我们使用示例类型系统来说明执行查询。这是这些文章中所有示例所使用的相同类型系统的一部分：

              type Query {
  human(id: ID!): Human
}

type Human {
  name: String
  appearsIn: [Episode]
  starships: [Starship]
}

enum Episode {
  NEWHOPE
  EMPIRE
  JEDI
}

type Starship {
  name: String
}

              为了描述执行查询时会发生什么，我们来看一个例子。

              {
  human (id: 1002) {
    name
    appearsIn
    starships {
      name
    }
  }
}

              {
  "data": {
    "human": {
      "name": "Han Solo"，
      "appearsIn": [
        为了描述执行查询时会发生什么，我们来看一个例子。
      ],
      "starships": [
        {
          "name": "Millenium Falcon"
        },
        {
          "name": "Imperial shuttle"
        }
      ]
    }
  }
}

              您可以将GraphQL查询中的每个字段视为返回下一个类型的上一个类型的函数或方法。实际上，这正是GraphQL的工作方式。每种类型的每个字段均由GraphQL服务器开发人员提供的称为解析器的功能支持。执行字段时，将调用相应的解析器以生成下一个值。

              如果字段产生标量值（例如字符串或数字），则执行完成。但是，如果字段产生对象值，则查询将包含适用于该对象的其他字段选择。这一直持续到达到标量值为止。 GraphQL查询始终以标量值结尾。

              # 根域和解析器

              在每个GraphQL服务器的顶层，都有一个表示GraphQL API的所有可能入口点的类型，通常称为Root类型或Query类型。

              在此示例中，我们的查询类型提供了一个名为human的字段，该字段接受参数id。该字段的解析器功能可能会访问数据库，然后构造并返回一个Human对象。

              Query: {
  human(obj, args, context, info) {
    return context.db.loadHumanById(args.id).then(
      userData => new Human(userData)
    )
  }
}

              这个例子是用JavaScript编写的，但是GraphQL服务器可以用许多不同的语言来构建。解析器函数接收四个参数：

              • obj先前的对象（根查询类型上的字段）通常不使用。
              • args提供给GraphQL查询中的字段的参数。
              • 上下文（context）一个值，提供给每个解析程序，并包含重要的上下文信息，例如当前登录的用户或对数据库的访问。
              • info一个值，它包含与当前查询有关的特定于字段的信息以及模式详细信息，有关更多详细信息，还请参考GraphQLResolveInfo类型。

              # 异步解析器

              让我们仔细看看此解析器函数中发生的情况。

              human (obj, args, context, info) {
  return context.db.loadHumanById(args.id).then(
    userData => new Human(userData)
  )
}

              上下文用于提供对数据库的访问，该数据库用于通过在GraphQL查询中作为参数提供的ID为用户加载数据。由于从数据库加载是异步操作，因此将返回Promise。在JavaScript中，Promises用于处理异步值，但是许多语言（通常称为Futures，Tasks或Deferred）中存在相同的概念。当数据库返回时，我们可以构造并返回一个新的Human对象。

              请注意，尽管解析程序功能需要了解Promises，但GraphQL查询却不需要。它只是希望人类领域返回某种东西，然后再询问它的名字。在执行期间，GraphQL将等待Promises，Future和Tasks完成后再继续执行，并以最佳的并发性完成。

              # 不重要的 resolvers

              既然Human对象可用，那么GraphQL的执行可以继续其上请求的字段。

              Human: {
  name(obj, args, context, info) {
    return obj.name
  }
}

              GraphQL服务器由类型系统提供动力，该类型系统用于确定下一步要做什么。甚至在人员返回任何内容之前，GraphQL都知道下一步将是解决人员类型上的字段，因为类型系统告诉它人员将返回人员。

              在这种情况下，解析名称非常简单。调用名称解析器函数，并且obj参数是从上一个字段返回的新的Human对象。在这种情况下，我们希望Human对象具有一个name属性，我们可以直接读取并返回它。

              实际上，许多GraphQL库会让您省略此解析器，而只是假设如果未为字段提供解析器，则应读取并返回相同名称的属性。

              # 标量强制

              在解析名称字段时，可以同时解析外观和星际飞船字段。 appearIn字段也可以有一个简单的解析器，但让我们仔细看一下：

              Human: {
  appearsIn (obj) {
    return obj.appearsIn
  }
}

              请注意，我们的类型系统claimsInformIn将返回具有已知值的Enum值，但是此函数将返回数字！确实，如果我们查看结果，将会看到返回了适当的Enum值。这是怎么回事？

              这是标量强制的一个例子。类型系统知道会发生什么，并将解析器函数返回的值转换为符合API约定的内容。在这种情况下，我们的服务器上可能定义了一个Enum，该Enum在内部使用像4、5和6这样的数字，但是在GraphQL类型系统中将它们表示为Enum值。

              # 列表解析器

              我们已经看到了当一个字段返回上面带有manifestIn字段的事物列表时发生的事情。它返回了一个枚举值列表，由于这是类型系统所期望的，因此列表中的每个项目都被强制为适当的枚举值。解决星际飞船字段后会发生什么？

              HUuman: {
  starships (obj, args, context, info) {
    return obj.starshipsIDs.map(
      id => context.db.loadStarshipByID(id).then(
        shipData => new Starship(shipData)
      )
    )
  }
}

              此字段的解析程序不仅返回一个Promise，而且还返回一个Promises列表。 Human对象具有他们驾驶的Starship的ID列表，但是我们需要加载所有这些ID才能获得真实的Starship对象。

              GraphQL将同时等待所有这些Promises，然后再继续，当留下对象列表时，它将再次并发继续以在每个项目上加载名称字段。

              # 产生结果

              解析每个字段时，将结果值放入以字段名（或别名）为键，并将解析值作为值的键值映射中，这从查询的底部叶字段一直到直到根查询类型上的原始字段。这些共同产生了一个镜像原始查询的结构，然后可以将其发送（通常为JSON）到请求它的客户端。

              让我们最后看一下原始查询，看看所有这些解析函数如何产生结果：

    name
    appearsIn
    starships {
      name
    }
  }
}
