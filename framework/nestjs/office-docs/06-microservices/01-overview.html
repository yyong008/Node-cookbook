# 总览

除了传统的（有时称为整体式）应用程序体系结构，Nest本机支持开发的微服务体系结构样式。本文档中其他地方讨论的大多数概念，例如依赖项注入，装饰器，异常过滤器，管道，防护和拦截器，同样适用于微服务。Nest会尽可能地抽象化实现细节，以便相同的组件可以在基于HTTP的平台，WebSocket和微服务上运行。本节涵盖了微服务特有的Nest方面。

在Nest中，微服务从根本上说是使用与HTTP 不同的传输层的应用程序。

Nest支持几种内置的传输层实现，称为传输器，负责在不同的微服务实例之间传输消息。大多数传输器本身都支持请求响应和基于事件的消息样式。Nest在规范接口的后面抽象了每个传输器的实现细节，用于请求响应和基于事件的消息传递。这样可以轻松地从一个传输层切换到另一层，例如，利用特定传输层的特定可靠性或性能功能，而不会影响您的应用程序代码。

# 安装

要开始构建微服务，请首先安装所需的软件包：

$ npm i --save @nestjs/microservices

# 入门

要实例化微服务，请使用类的createMicroservice()方法NestFactory：

import { NestFactory } from '@nestjs/core';
import { Transport, MicroserviceOptions } from '@nestjs/microservices';
import { AppModule } from './app.module';

async function bootstrap() {
  const app = await NestFactory.createMicroservice<MicroserviceOptions>(
    AppModule,
    {
      transport: Transport.TCP,
    },
  );
  app.listen(() => console.log('Microservice is listening'));
}
bootstrap();

::tip 提示微服务 默认情况下使用TCP传输层。 :::

该createMicroservice()方法的第二个参数是一个options对象。该对象可能包含两个成员：

transport 指定运输者（例如，Transport.NATS） options 特定于运输者的选项对象，确定运输者的行为

该options对象特定于所选的运输者。的TCP转运暴露性质如下所述。对于其他运输工具（例如Redis，MQTT等），请参阅相关章节以获取可用选项的说明。

host 连接主机名 port 连接口 retryAttempts 次地重试消息（默认值：0） retryDelay （ms）的消息重试之间的延迟（默认0）

# 模式

微服务认识到这两个消息和事件模式。模式是一个普通值，例如，文字对象或字符串。模式将自动序列化，并与消息的数据部分一起通过网络发送。通过这种方式，消息发送者和使用者可以协调哪些请求由哪些处理程序使用。

# 请求-响应

当您需要在各种外部服务之间交换消息时，请求-响应消息样式很有用。使用此范例，您可以确定该服务实际上已经收到了该消息（无需手动实现消息ACK协议）。但是，请求-响应范例并不总是最佳选择。例如，对使用基于日志的持久性的流传输器（例如Kafka或NATS流）进行了优化，以解决不同范围的问题，更符合事件消息传递范式（有关更多详细信息，请参见下面的基于事件的消息传递）。

为了启用请求-响应消息类型，Nest创建了两个逻辑通道-一个负责传输数据，而另一个则等待传入的响应。对于某些基础传输，例如NATS，此双通道支持是开箱即用的。对于其他人，Nest通过手动创建单独的渠道进行补偿。这样做可能会有开销，因此，如果您不需要请求-响应消息样式，则应考虑使用基于事件的方法。

要基于请求-响应范例创建消息处理程序，请使用@MessagePattern()从@nestjs/microservices包中导入的装饰器。

import { Controller } from '@nestjs/common';
import { MessagePattern } from '@nestjs/microservices';

@Controller()
export class MathController {
  @MessagePattern({ cmd: 'sum' })
  accumulate(data: number[]): number {
    return (data || []).reduce((a, b) => a + b);
  }
}

在上面的代码中，accumulate()消息处理程序侦听满足{ cmd: 'sum' }消息模式的消息。消息处理程序采用单个参数，即data从客户端传递的参数。在这种情况下，数据是要累积的数字数组。

# 异步响应

消息处理程序能够同步或异步响应。因此，async支持方法。

@MessagePattern({ cmd: 'sum' })
async accumulate(data: number[]): Promise<number> {
  return (data || []).reduce((a, b) => a + b);
}

消息处理程序还可以返回 Observable，在这种情况下，将发射结果值，直到流完成为止。

@MessagePattern({ cmd: 'sum' })
accumulate(data: number[]): Observable<number> {
  return from([1, 2, 3]);
}

在上面的示例中，消息处理程序将响应3次（数组中的每个项）。

# 基于事件

虽然request-response方法是在服务之间交换消息的理想方法，但是当您的消息样式基于事件时（即您只想发布事件而不等待响应时），它不太适合。在那种情况下，您不希望请求响应维护两个通道所需的开销。

假设您只想简单地通知另一服务在系统的这一部分中已经发生了某种情况。这是基于事件的消息样式的理想用例。

要创建事件处理程序，我们使用@EventPattern()装饰器，该装饰器是从@nestjs/microservices包中导入的。

@EventPattern('user_created')
async handleUserCreated(data: Record<string, unknown>) {
  // business logic
}

该handleUserCreated()事件处理程序侦听'user_created'事件。事件处理程序采用单个参数，即data从客户端传递的参数（在这种情况下，是已通过网络发送的事件有效负载）。

# 修饰器

在更复杂的方案中，您可能需要访问有关传入请求的更多信息。例如，对于带有通配符订阅的NATS，您可能想要获取生产者已将消息发送到的原始主题。同样，在Kafka中，您可能要访问消息头。为此，可以使用内置装饰器，如下所示：

@MessagePattern('time.us.*')
getDate(@Payload() data: number[], @Ctx() context: NatsContext) {
  console.log(`Subject: ${context.getSubject()}`); // e.g. "time.us.east"
  return new Date().toLocaleTimeString(...);
}

TIP

提示 @Payload()，@Ctx()并且NatsContext是从导入的@nestjs/microservices。

# 客户

客户端Nest应用程序可以使用ClientProxy该类交换消息或将事件发布到Nest微服务。此类定义了几种方法，例如send()（用于请求-响应消息传递）和emit()（用于事件驱动的消息传递），可让您与远程微服务进行通信。通过以下方式之一获取此类的实例。

一种技术是导入ClientsModule，它公开了静态register()方法。此方法采用一个参数，该参数是表示微服务传输程序的对象数组。每个此类对象都有一个name属性，一个可选transport属性（默认为Transport.TCP）和一个可选的特定于运输车的options属性。

该name属性用作注入令牌，可用于ClientProxy在需要的地方注入实例。所述的值name属性，作为注射剂的道理，可以是任意的字符串或JavaScript符号，如所描述这里。

该options属性是一个对象，具有与createMicroservice()前面方法中看到的属性相同的对象。

@Module({
  imports: [
    ClientsModule.register([
      { name: 'MATH_SERVICE', transport: Transport.TCP },
    ]),
  ]
  ...
})

导入模块后，我们可以使用装饰器注入ClientProxy通过上述'MATH_SERVICE'显示的传输器选项指定的已配置实例@Inject()

constructor(
  @Inject('MATH_SERVICE') private client: ClientProxy,
) {}

::: 提示 本ClientsModule和ClientProxy类是从进口@nestjs/microservices包。 :::

有时我们可能需要从另一个服务（例如ConfigService）中获取传输器配置，而不是在我们的客户端应用程序中对其进行硬编码。为此，我们可以使用该类注册自定义提供程序ClientProxyFactory。此类具有静态create()方法，该方法接受运输工具选项对象，并返回自定义ClientProxy实例。

@Module({
  providers: [
    {
      provide: 'MATH_SERVICE',
      useFactory: (configService: ConfigService) => {
        const mathSvcOptions = configService.getMathSvcOptions();
        return ClientProxyFactory.create(mathSvcOptions);
      },
      inject: [ConfigService],
    }
  ]
  ...
})

TIP

提示 ClientProxyFactory从进口@nestjs/microservices包。

另一种选择是使用@Client()属性装饰器。

@Client({ transport: Transport.TCP })
client: ClientProxy;

TIP

提示 该@Client()装饰器从导入的@nestjs/microservices包。

使用@Client()装饰器不是首选技术，因为它很难测试，也很难共享客户端实例。

该ClientProxy是懒惰的。它不会立即启动连接。相反，它将在第一个微服务调用之前建立，然后在每个后续调用中重用。但是，如果您希望将应用程序引导过程延迟到建立连接之前，则可以使用生命周期挂钩内的ClientProxy对象的connect()方法手动启动连接OnApplicationBootstrap。

async onApplicationBootstrap() {
  await this.client.connect();
}

如果无法创建连接，则该connect()方法将拒绝并带有相应的错误对象。

# 传送讯息

该ClientProxy公开了一个send()方法。该方法旨在调用微服务并返回Observable带有其响应的。因此，我们可以轻松地订阅发射的值。

accumulate(): Observable<number> {
  const pattern = { cmd: 'sum' };
  const payload = [1, 2, 3];
  return this.client.send<number>(pattern, payload);
}

该send()方法有两个参数，pattern和payload。本pattern应与所定义的一个@MessagePattern()装饰。这payload是我们要传输到远程微服务的消息。此方法返回ColdObservable，这意味着您必须显式订阅它才能发送消息。

# 发布事件

要发送事件，请使用ClientProxy对象的emit()方法。此方法将事件发布到消息代理。

async publish() {
  this.client.emit<number>('user_created', new UserCreatedEvent());
}

该emit()方法有两个参数，pattern和payload。本pattern应与在定义一个@EventPattern()装饰。这payload是我们要传输到远程微服务的事件有效负载。此方法返回一个热点Observable（与所Observable返回的感冒不同send()），这意味着无论您是否显式预订可观察对象，代理都将立即尝试传递事件。

# 范围

对于来自不同编程语言背景的人们来说，意外地发现在Nest中几乎所有内容都是在传入请求之间共享的。我们有到数据库的连接池，具有全局状态的单例服务等。请记住，Node.js并不遵循请求/响应多线程无状态模型，在该模型中，每个请求都由单独的线程处理。因此，使用单例实例对于我们的应用程序是完全安全的。

但是，在某些情况下，处理程序的基于请求的生存期可能是所需的行为，例如GraphQL应用程序中的每个请求缓存，请求跟踪或多租户。在此处了解如何控制范围。

请求范围的处理程序和提供程序可以结合RequestContext使用@Inject()装饰器和CONTEXT令牌进行注入：

import { Injectable, Scope, Inject } from '@nestjs/common';
import { CONTEXT, RequestContext } from '@nestjs/microservices';

@Injectable({ scope: Scope.REQUEST })
export class CatsService {
  constructor(@Inject(CONTEXT) private ctx: RequestContext) {}
}

这提供了对RequestContext对象的访问，该对象具有两个属性：

export interface RequestContext<T = any> {
  pattern: string | Record<string, any>;
  data: T;
}x

该data属性是消息生产者发送的消息有效负载。该pattern属性是用于标识适当的处理程序以处理传入消息的模式。
