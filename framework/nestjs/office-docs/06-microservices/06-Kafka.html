# Kafka

Kafka是一个开放源代码的分布式流媒体平台，具有三个关键功能：

发布和订阅记录流，类似于消息队列或企业消息传递系统。 以容错的持久方式存储记录流。 处理记录流。 Kafka项目旨在提供一个统一的，高吞吐量，低延迟的平台来处理实时数据馈送。它与Apache Storm和Spark集成得很好，可以进行实时流数据分析。

Kafka 转运 是实验性的。

# 安装

要开始构建基于Kafka的微服务，请首先安装所需的软件包：

npm i --save kafkajs

# 概述

与其他Nest微服务传输层实现一样，您可以使用transport传递给该createMicroservice()方法的options对象的属性以及一个可选options属性来选择Kafka传输器机制，如下所示：

const app = await NestFactory.createMicroservice<MicroserviceOptions>(ApplicationModule, {
  transport: Transport.KAFKA,
  options: {
    client: {
      brokers: ['localhost:9092'],
    }
  }
});

TIP

提示 该Transport枚举从导入的@nestjs/microservices包。

# 选项

该options属性特定于所选的运输者。所述卡夫卡转运暴露性质如下所述。

1 2
client 客户端配置选项（在此处了解更多信息）
consumer 使用者配置选项（在此处了解更多信息）
run 运行配置选项（在此处了解更多信息）
subscribe 订阅配置选项（在此处了解更多信息）
producer 生产者配置选项（在此处了解更多信息）
send 发送配置选项（在此处了解更多信息）

# 客户

与其他微服务运输商相比，Kafka差异很小。代替ClientProxy类，我们使用ClientKafka类。

与其他微服务传输器一样，您可以通过多种方法来创建ClientKafka实例。

创建实例的一种方法是使用ClientsModule。要使用创建一个客户端实例ClientsModule，请导入该实例，并使用register()方法传递一个选项对象，该对象具有与createMicroservice()方法中上面显示的相同的属性，以及用作name注入令牌的属性。ClientsModule在此处了解更多信息。

@Module({
  imports: [
    ClientsModule.register([
      {
        name: 'HERO_SERVICE',
        transport: Transport.KAFKA,
        options: {
          client: {
            clientId: 'hero',
            brokers: ['localhost:9092'],
          },
          consumer: {
            groupId: 'hero-consumer'
          }
        }
      },
    ]),
  ]
  ...
})

也可以使用其他创建客户端的选项（ClientProxyFactory或@Client()）。您可以在这里阅读有关它们的信息。

使用@Client()装饰器，如下所示：

@Client({
  transport: Transport.KAFKA,
  options: {
    client: {
      clientId: 'hero',
      brokers: ['localhost:9092'],
    },
    consumer: {
      groupId: 'hero-consumer'
    }
  }
})
client: ClientKafka;

# 消息响应订阅

ClientKafka类提供的subscribeToResponseOf()方法。该subscribeToResponseOf()方法将请求的主题名称作为参数，并将派生的答复主题名称添加到答复主题的集合中。实现消息模式时，此方法是必需的。

onModuleInit() {
  this.client.subscribeToResponseOf('hero.kill.dragon');
}

如果ClientKafka实例是异步创建的，则subscribeToResponseOf()必须在调用connect()方法之前调用该方法。

async onModuleInit() {
  this.client.subscribeToResponseOf('hero.kill.dragon');
  await this.client.connect();
}

# 消息模式

Kafka微服务消息模式将两个主题用于请求和答复通道。该ClientKafka#send()方法通过将相关ID，回复主题和回复分区与请求消息相关联来发送带有返回地址的消息。这要求实例在发送消息之前先订阅回复主题并分配给至少一个分区。ClientKafka

随后，您需要为每个运行的Nest应用程序至少有一个答复主题分区。例如，如果您正在运行4个Nest应用程序，但答复主题只有3个分区，则尝试发送消息时，其中1个Nest应用程序将出错。

ClientKafka启动新实例后，它们将加入消费者组并订阅各自的主题。此过程触发分配给使用者组使用者的主题分区的重新平衡。

通常，主题分区是使用循环分区程序分配的，该程序将主题分区分配给按消费者名称排序的消费者集合，消费者名称是在应用程序启动时随机设置的。但是，当新消费者加入该消费者组时，该新消费者可以位于消费者集合中的任何位置。这就创造了这样一种条件，即当现有消费者位于新消费者之后时，可以为现有消费者分配不同的分区。结果，分配了不同分区的使用者将丢失重新平衡之前发送的请求的响应消息。

为了防止ClientKafka使用者丢失响应消息，使用了Nest特定的内置自定义分区程序。该自定义分区程序将分区分配给使用者的集合，这些使用者按照process.hrtime()在应用程序启动时设置的高分辨率时间戳（）进行排序。

# 传入

窝接收传入消息卡夫卡作为与对象key，value和headers具有类型的值的属性Buffer。然后，Nest通过将缓冲区转换为字符串来解析这些值。如果字符串是“ like like”，则Nest尝试将字符串解析为JSON。在value随后被传递到其相关的处理程序。

# 外发

在发布事件或发送消息时，Nest在序列化过程之后发送传出的Kafka消息。这会在传递给ClientKafkaemit()和send()方法的参数或方法返回的值上发生@MessagePattern。通过使用JSON.stringify()或toString()原型方法，此序列化可将不是字符串或缓冲区的对象“字符串化” 。

@Controller()
export class HeroController {
  @MessagePattern('hero.kill.dragon')
  killDragon(@Payload() message: KillDragonMessage): any {
    const dragonId = message.dragonId;
    const items = [
      { id: 1, name: 'Mythical Sword' },
      { id: 2, name: 'Key to Dungeon' },
    ];
    return items;
  }
}

TIP

提示 @Payload()是从导入的@nestjs/microservices。

传出的消息也可以通过传递带有key和value属性的对象来键入。键控消息对于满足共分区要求很重要。

@Controller()
export class HeroController {
  @MessagePattern('hero.kill.dragon')
  killDragon(@Payload() message: KillDragonMessage): any {
    const realm = 'Nest';
    const heroId = message.heroId;
    const dragonId = message.dragonId;

    const items = [
      { id: 1, name: 'Mythical Sword' },
      { id: 2, name: 'Key to Dungeon' },
    ];

    return {
      headers: {
        realm
      },
      key: heroId,
      value: items
    }
  }
}

此外，以这种格式传递的邮件还可以包含在headershash属性中设置的自定义标头。标头哈希属性值必须为type string或type Buffer。

@Controller()
export class HeroController {
  @MessagePattern('hero.kill.dragon')
  killDragon(@Payload() message: KillDragonMessage): any {
    const realm = 'Nest';
    const heroId = message.heroId;
    const dragonId = message.dragonId;

    const items = [
      { id: 1, name: 'Mythical Sword' },
      { id: 2, name: 'Key to Dungeon' },
    ];

    return {
      headers: {
        kafka_nestRealm: realm
      },
      key: heroId,
      value: items
    }
  }
}

# 上下文

在更复杂的方案中，您可能需要访问有关传入请求的更多信息。使用Kafka传输器时，可以访问该KafkaContext对象。

@MessagePattern('hero.kill.dragon')
killDragon(@Payload() message: KillDragonMessage, @Ctx() context: KafkaContext) {
  console.log(`Topic: ${context.getTopic()}`);
}

TIP

提示 @Payload()，@Ctx()并且KafkaContext从@nestjs/microservices包中导入。

要访问原始的Kafka IncomingMessage对象，请使用对象的getMessage()方法KafkaContext，如下所示：

@MessagePattern('hero.kill.dragon')
killDragon(@Payload() message: KillDragonMessage, @Ctx() context: KafkaContext) {
  const originalMessage = context.getMessage();
  const { headers, partition, timestamp } = originalMessage;
}

凡IncomingMessage满足以下界面：

interface IncomingMessage {
  topic: string;
  partition: number;
  timestamp: string;
  size: number;
  attributes: number;
  offset: string;
  key: any;
  value: any;
  headers: Record<string, any>;
}

# 名字约定

Kafka微服务组件将其各自角色的描述附加到client.clientId和consumer.groupId选项上，以防止Nest微服务客户端和服务器组件之间发生冲突。默认情况下，ClientKafka组件会附加-client，而ServerKafka组件会附加-server到这两个选项。请注意，下面提供的值是如何以这种方式转换的（如注释中所示）。

const app = await NestFactory.createMicroservice(ApplicationModule, {
  transport: Transport.KAFKA,
  options: {
    client: {
      clientId: 'hero', // hero-server
      brokers: ['localhost:9092'],
    },
    consumer: {
      groupId: 'hero-consumer' // hero-consumer-server
    },
  }
});

对于客户：

@Client({
  transport: Transport.KAFKA,
  options: {
    client: {
      clientId: 'hero', // hero-client
      brokers: ['localhost:9092'],
    },
    consumer: {
      groupId: 'hero-consumer' // hero-consumer-client
    }
  }
})
client: ClientKafka;

TIP

提示 卡夫卡客户和消费者的命名约定可以通过扩展定制ClientKafka并KafkaServer在自己的自定义提供商，重写构造函数。

由于Kafka微服务消息模式将两个主题用于请求和回复通道，因此应从请求主题中得出一个回复模式。默认情况下，回复主题的名称是请求主题名称加.reply后缀的组合。

onModuleInit() {
  this.client.subscribeToResponseOf('hero.get'); // hero.get.reply
}

TIP

可以通过扩展ClientKafka您自己的自定义提供程序并覆盖该getResponsePatternName方法来自定义 提示 Kafka答复主题命名约定。
