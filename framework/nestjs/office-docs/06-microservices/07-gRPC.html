# gRPC

gRPC是可以在任何环境中运行的现代，开源，高性能RPC框架。它可插拔支持负载平衡，跟踪，运行状况检查和身份验证，从而可以有效地连接数据中心内和跨数据中心的服务。

像许多RPC系统一样，gRPC的概念是根据可以远程调用的功能（方法）来定义服务。对于每种方法，您都可以定义参数和返回类型。服务，参数和返回类型是.proto使用Google的开源语言中立协议缓冲区机制在文件中定义的。

借助gRPC传输器，Nest使用.proto文件动态绑定客户端和服务器，从而轻松实现远程过程调用，自动序列化和反序列化结构化数据。

# 安装

要开始构建基于gRPC的微服务，请首先安装所需的软件包：

npm i --save grpc @grpc/proto-loader

# 总览

与其他Nest微服务传输层实现一样，您可以使用transport传递给createMicroservice()方法的options对象的属性来选择gRPC传输器机制。在以下示例中，我们将设置英雄服务。该options属性提供有关该服务的元数据。其性能描述如下。

const app = await NestFactory.createMicroservice<MicroserviceOptions>(AppModule, {
  transport: Transport.GRPC,
  options: {
    package: 'hero',
    protoPath: join(__dirname, 'hero/hero.proto'),
  },
});

TIP

提示 该join()功能是从path包中导入的；该Transport枚举从导入的@nestjs/microservices包。

# gRPC 服务样本

让我们定义名为gRPC的示例服务HeroService。在上述options对象中，该protoPath属性设置了一个指向.protodefinitions文件的路径hero.proto。该hero.proto文件是使用协议缓冲区构造的。看起来是这样的：

// hero/hero.proto
syntax = "proto3";

package hero;

service HeroService {
  rpc FindOne (HeroById) returns (Hero) {}
}

message HeroById {
  int32 id = 1;
}

message Hero {
  int32 id = 1;
  string name = 2;
}

我们HeroService公开了一种FindOne()方法。该方法期望输入参数为type HeroById并返回一条Hero消息（协议缓冲区使用message元素来定义参数类型和返回类型）。

接下来，我们需要实现服务。要定义满足此定义的处理程序，我们@GrpcMethod()在控制器中使用装饰器，如下所示。该装饰器提供将方法声明为gRPC服务方法所需的元数据。

:::tips 提示 在先前的微服务章节中介绍的@MessagePattern()装饰器（阅读更多内容）不适用于基于gRPC的微服务。该@GrpcMethod()装饰有效地采取了基于GRPC-微服务到位。 :::

@Controller()
export class HeroController {
  @GrpcMethod('HeroService', 'FindOne')
  findOne(data: HeroById, metadata: any): Hero {
    const items = [
      { id: 1, name: 'John' },
      { id: 2, name: 'Doe' },
    ];
    return items.find(({ id }) => id === data.id);
  }
}

TIP

提示 该@GrpcMethod()装饰器从导入的@nestjs/microservices包。

上面显示的装饰器带有两个参数。第一个是服务名称（例如'HeroService'），与中的HeroService服务定义相对应hero.proto。第二个（字符串'FindOne'）对应于文件中FindOne()定义的rpc方法。HeroServicehero.proto

该findOne()处理程序方法采用两个参数，即data从调用方传递的参数，该参数metadata存储gRPC请求的元数据。

这两个@GrpcMethod()装饰器参数都是可选的。如果在没有第二个参数（例如'FindOne'）的情况下调用，Nest将.proto基于将处理程序名称转换为大写驼峰（例如，findOne处理程序与FindOnerpc调用定义相关联），自动将文件rpc方法与处理程序相关联。如下所示。

@Controller()
export class HeroController {
  @GrpcMethod('HeroService')
  findOne(data: HeroById, metadata: any): Hero {
    const items = [
      { id: 1, name: 'John' },
      { id: 2, name: 'Doe' },
    ];
    return items.find(({ id }) => id === data.id);
  }
}

您也可以省略第一个@GrpcMethod()参数。在这种情况下，Nest会根据定义处理程序的类名称自动将处理程序与原始定义文件中的服务定义相关联。例如，在以下代码中，类基于name的匹配将HeroService其处理程序方法与文件中的HeroService服务定义相关联。hero.proto'HeroService'

@Controller()
export class HeroService {
  @GrpcMethod()
  findOne(data: HeroById, metadata: any): Hero {
    const items = [
      { id: 1, name: 'John' },
      { id: 2, name: 'Doe' },
    ];
    return items.find(({ id }) => id === data.id);
  }
}

# 客户

Nest应用程序可以充当gRPC客户端，使用.proto文件中定义的服务。您通过ClientGrpc对象访问远程服务。您可以通过ClientGrpc多种方式获取对象。

首选技术是导入ClientsModule。使用该register()方法可以将.proto文件中定义的服务包绑定到注入令牌，并配置服务。该name属性是注入令牌。对于gRPC服务，请使用transport: Transport.GRPC。所述options属性是与所述相同的属性的对象之上。

imports: [
  ClientsModule.register([
    {
      name: 'HERO_PACKAGE',
      transport: Transport.GRPC,
      options: {
        package: 'hero',
        protoPath: join(__dirname, 'hero/hero.proto'),
      },
    },
  ]),
];

TIP

提示 该register()方法接受一个对象数组。通过提供逗号分隔的注册对象列表来注册多个软件包。

注册后，我们可以使用注入配置的ClientGrpc对象@Inject()。然后，我们使用ClientGrpc对象的getService()方法来检索服务实例，如下所示。

@Injectable()
export class AppService implements OnModuleInit {
  private heroService: HeroService;

  constructor(@Inject('HERO_PACKAGE') private client: ClientGrpc) {}

  onModuleInit() {
    this.heroService = this.client.getService<HeroService>('HeroService');
  }

  getHero(): Observable<string> {
    return this.heroService.findOne({ id: 1 });
  }
}

请注意，与其他微服务传输方法中使用的技术相比，差异很小。代替ClientProxy类，我们使用ClientGrpc提供getService()方法的类。该getService()通用方法将一个服务名作为参数，并返回其实例（如果可用）。

或者，可以使用@Client()装饰器实例化一个ClientGrpc对象，如下所示：

@Injectable()
export class AppService implements OnModuleInit {
  @Client({
    transport: Transport.GRPC,
    options: {
      package: 'hero',
      protoPath: join(__dirname, 'hero/hero.proto'),
    },
  })
  client: ClientGrpc;

  private heroService: HeroService;

  onModuleInit() {
    this.heroService = this.client.getService<HeroService>('HeroService');
  }

  getHero(): Observable<string> {
    return this.heroService.findOne({ id: 1 });
  }
}

最后，对于更复杂的场景，我们可以使用此处ClientProxyFactory介绍的类注入动态配置的客户端。

无论哪种情况，我们都以对HeroService代理对象的引用结尾，该引用暴露了.proto文件内部定义的相同方法集。现在，当我们访问该代理对象（即heroService）时，gRPC系统会自动序列化请求，将其转发到远程系统，返回响应，然后反序列化响应。因为gRPC使我们免受这些网络通信细节的影响，所以heroService外观和行为类似于本地提供程序。

请注意，所有服务方法都是小写驼峰式的（为了遵循语言的自然习惯）。因此，例如，当我们的.proto文件HeroService定义包含FindOne()函数时，heroService实例将提供该findOne()方法。

interface HeroService {
  findOne(data: { id: number }): Observable<any>;
}

消息处理程序还可以返回 Observable，在这种情况下，将发射结果值，直到流完成为止。

@Get()
call(): Observable<any> {
  return this.heroService.findOne({ id: 1 });
}

# gRPC

gRPC本身支持长期的实时连接，通常称为streams。对于诸如聊天，观察或块数据传输之类的情况，流非常有用。在此处找到官方文档中的更多详细信息。

Nest通过两种可能的方式支持GRPC流处理程序：

RxJS Subject+ Observable处理程序：在控制器方法内部编写响应或向下传递给Subject/ Observable消费者很有用 纯GRPC调用流处理程序：可以将其传递给某些执行程序，该执行程序将处理Node标准Duplex流处理程序的其余分派。

# 流样本

让我们定义一个名为的新示例gRPC服务HelloService。该hello.proto文件是使用协议缓冲区构造的。看起来是这样的：

// hello/hello.proto
syntax = "proto3";

package hello;

service HelloService {
  rpc BidiHello(stream HelloRequest) returns (stream HelloResponse)
  rpc LotsOfGreetings(stream HelloRequest) returns (HelloResponse)
}

message HelloRequest {
  string greeting = 1;
}

message HelloResponse {
  string reply = 1;
}

TIP

提示 由于返回的流可以发出多个值，因此可以LotsOfGreetings使用@GrpcMethod装饰器轻松实现该方法（如上述示例中所示）。

基于此.proto文件，让我们定义HelloService接口：

interface HelloService {
  bidiHello(upstream: Observable<HelloRequest>): Observable<HelloResponse>;
  lotsOfGreetings(
    upstream: Observable<HelloRequest>,
  ): Observable<HelloResponse>;
}

interface HelloRequest {
  greeting: string;
}

interface HelloResponse {
  reply: string;
}

# 学科策略

@GrpcStreamMethod()装饰提供了函数参数作为RxJS Observable。因此，我们可以接收和处理多个消息。

@GrpcStreamMethod()
bidiHello(messages: Observable<any>): Observable<any> {
  const subject = new Subject();

  const onNext = message => {
    console.log(message);
    subject.next({
      reply: 'Hello, world!'
    });
  };
  const onComplete = () => subject.complete();
  messages.subscribe(onNext, null, onComplete);

  return subject.asObservable();
}

TIP

提示 为了支持与@GrpcStreamMethod()装饰器的全双工交互，controller方法必须返回RxJS Observable。

根据服务定义（在.proto文件中），该BidiHello方法应将请求流式传输到服务。要从客户端向流发送多个异步消息，我们利用RxJS ReplySubject类。

const helloService = this.client.getService<HelloService>('HelloService');
const helloRequest$ = new ReplySubject<HelloRequest>();

helloRequest$.next({ greeting: 'Hello (1)!' });
helloRequest$.next({ greeting: 'Hello (2)!' });
helloRequest$.complete();

return helloService.bidiHello(helloRequest$);

在上面的示例中，我们向流（next()调用）中写入了两条消息，并通知服务我们已经完成了数据（complete()调用）的发送。

# 呼叫流处理程序

当该方法返回值被定义为stream，所述@GrpcStreamCall()装饰提供功能参数如grpc.ServerDuplexStream，支持的标准方法等.on('data', callback)，.write(message)或.cancel()。有关可用方法的完整文档，请参见此处。

可替代地，当该方法返回值不是一个stream，所述@GrpcStreamCall()装饰提供了两个函数的参数，分别grpc.ServerReadableStream（阅读更多这里）和callback。

让我们从实现BidiHello应该支持全双工交互的开始。

@GrpcStreamCall()
bidiHello(requestStream: any) {
  requestStream.on('data', message => {
    console.log(message);
    requestStream.write({
      reply: 'Hello, world!'
    });
  });
}

TIP

提示 此修饰器不需要提供任何特定的返回参数。期望该流将以与任何其他标准流类型类似的方式处理。

在上面的示例中，我们使用了write()将对象写入响应流的方法。.on()每当我们的服务接收到新的数据块时，将调用作为第二个参数传入该方法的回调。

让我们实现该LotsOfGreetings方法。

@GrpcStreamCall()
lotsOfGreetings(requestStream: any, callback: (err: unknown, value: HelloResponse) => void) {
  requestStream.on('data', message => {
    console.log(message);
  });
  requestStream.on('end', () => callback(null, { reply: 'Hello, world!' }));
}
