# 测试中

自动化测试被视为任何认真的软件开发工作的重要组成部分。自动化使得在开发过程中快速，轻松地重复单个测试或测试套件变得容易。这有助于确保发行版符合质量和性能目标。自动化有助于扩大覆盖范围，并为开发人员提供更快的反馈循环。自动化既可以提高单个开发人员的生产率，又可以确保测试在关键的开发生命周期关头运行，例如源代码控制检入，功能集成和版本发布。

此类测试通常涉及多种类型，包括单元测试，端到端（e2e）测试，集成测试等等。虽然好处是毋庸置疑的，但设置这些好处可能很繁琐。Nest致力于促进开发最佳实践，包括有效的测试，因此它包含以下功能，以帮助开发人员和团队构建和自动化测试。巢：

自动为组件提供默认的单元测试和为应用程序提供端到端测试 提供默认工具（例如构建独立模块/应用程序加载器的测试运行器） 开箱即用地提供与Jest和Supertest的集成，同时与测试工具无关 使Nest依赖项注入系统可在测试环境中使用，以轻松模拟组件 如前所述，您可以使用自己喜欢的任何测试框架，因为Nest不会强制使用任何特定工具。只需替换所需的元素（例如测试运行程序），您仍将享受Nest的现成测试设施的好处。

# 安装

首先，请先安装所需的软件包：

$ npm i --save-dev @nestjs/testing

# 单元测试

在以下示例中，我们测试了两个类：CatsController和CatsService。如前所述，Jest被提供为默认测试框架。它充当测试运行器，还提供断言函数和双重测试实用程序，以帮助进行模拟，监视等。在下面的基本测试中，我们手动实例化这些类，并确保控制器和服务履行其API约定。

import { CatsController } from './cats.controller';
import { CatsService } from './cats.service';

describe('CatsController', () => {
  let catsController: CatsController;
  let catsService: CatsService;

  beforeEach(() => {
    catsService = new CatsService();
    catsController = new CatsController(catsService);
  });

  describe('findAll', () => {
    it('should return an array of cats', async () => {
      const result = ['test'];
      jest.spyOn(catsService, 'findAll').mockImplementation(() => result);

      expect(await catsController.findAll()).toBe(result);
    });
  });
});

TIP

提示 将您的测试文件 放置在 要测试的类附近。测试文件应带有.spec或.test后缀

由于上面的示例是微不足道的，因此我们并未真正测试任何特定于Nest的内容。确实，我们甚至没有使用依赖注入（注意，我们将实例传递CatsService给catsController）。这种测试形式-我们手动实例化要测试的类-通常被称为隔离测试，因为它独立于框架。让我们介绍一些更高级的功能，这些功能可以帮助您测试更广泛使用Nest功能的应用程序。

# 测试实用程序

该@nestjs/testing软件包提供了一组实用程序，可以使测试过程更可靠。让我们使用内置Test类重写前面的示例.

import { Test } from '@nestjs/testing';
import { CatsController } from './cats.controller';
import { CatsService } from './cats.service';

describe('CatsController', () => {
  let catsController: CatsController;
  let catsService: CatsService;

  beforeEach(async () => {
    const moduleRef = await Test.createTestingModule({
        controllers: [CatsController],
        providers: [CatsService],
      }).compile();

    catsService = moduleRef.get<CatsService>(CatsService);
    catsController = moduleRef.get<CatsController>(CatsController);
  });

  describe('findAll', () => {
    it('should return an array of cats', async () => {
      const result = ['test'];
      jest.spyOn(catsService, 'findAll').mockImplementation(() => result);

      expect(await catsController.findAll()).toBe(result);
    });
  });
});

本Test类是提供基本上嘲笑满巢运行的应用程序执行环境有益，但给你钩，可以轻松管理类的实例，包括嘲弄和覆盖。的Test类有一个createTestingModule()接受一个模块元数据对象作为其参数（将传递给同一对象方法@Module()装饰）。此方法返回一个TestingModule实例，该实例又提供了一些方法。对于单元测试，重要的是compile()方法。此方法使用其依赖项引导模块（类似于main.ts使用常规文件引导应用程序的方式NestFactory.create()），并返回准备进行测试的模块。

TIP

提示 该compile()方法是异步的，因此必须等待。编译模块后，您可以使用方法检索它声明的任何静态实例（控制器和提供程序）get()

TestingModule继承自模块引用类，因此具有动态解析作用域提供者（瞬态或请求范围的）的能力。使用resolve()方法执行此操作（该get()方法只能检索静态实例）。

const moduleRef = await Test.createTestingModule({
  controllers: [CatsController],
  providers: [CatsService],
}).compile();

catsService = await moduleRef.resolve(CatsService);

TIP

警告 该resolve()方法从其自己的DI容器子树中返回提供者的唯一实例。每个子树都有一个唯一的上下文标识符。因此，如果多次调用此方法并比较实例引用，您会发现它们不相等。

WARNING

提示 在此处了解有关模块参考功能的更多信息。

除了使用任何提供程序的生产版本，您还可以使用自定义提供程序覆盖它以进行测试。例如，您可以模拟数据库服务而不是连接到实时数据库。我们将在下一部分中介绍替代，但是它们也可用于单元测试。

# 终端到终端的测试

与侧重于单个模块和类的单元测试不同，端到端（e2e）测试涵盖了更汇总级别的类和模块的交互-更接近最终用户与产品之间的交互类型。系统。随着应用程序的增长，很难手动测试每个API端点的端到端行为。自动化的端到端测试可帮助我们确保系统的整体行为正确并符合项目要求。为了执行e2e测试，我们使用与刚才在单元测试中介绍的配置类似的配置。此外，Nest可以轻松使用Supertest库来模拟HTTP请求。

import * as request from 'supertest';
import { Test } from '@nestjs/testing';
import { CatsModule } from '../../src/cats/cats.module';
import { CatsService } from '../../src/cats/cats.service';
import { INestApplication } from '@nestjs/common';

describe('Cats', () => {
  let app: INestApplication;
  let catsService = { findAll: () => ['test'] };

  beforeAll(async () => {
    const moduleRef = await Test.createTestingModule({
      imports: [CatsModule],
    })
      .overrideProvider(CatsService)
      .useValue(catsService)
      .compile();

    app = moduleRef.createNestApplication();
    await app.init();
  });

  it(`/GET cats`, () => {
    return request(app.getHttpServer())
      .get('/cats')
      .expect(200)
      .expect({
        data: catsService.findAll(),
      });
  });

  afterAll(async () => {
    await app.close();
  });
});

在此示例中，我们以前面描述的一些概念为基础。除了compile()我们先前使用的createNestApplication()方法之外，我们现在还使用该方法实例化完整的Nest运行时环境。我们将对正在运行的应用程序的引用保存在app变量中，以便可以使用它来模拟HTTP请求。

我们使用request()Supertest中的函数模拟HTTP测试。我们希望这些HTTP请求路由到我们正在运行的Nest应用程序，因此我们向该request()函数传递了对Nest之下的HTTP侦听器的引用（而该侦听器又可以由Express平台提供）。因此施工request(app.getHttpServer())。现在，request()我们将调用包装后的HTTP Server（现在已连接到Nest应用程序）的调用，该服务器公开了模拟实际HTTP请求的方法。例如，使用request(...).get('/cats')会向Nest应用发起一个与实际 HTTP请求相同的请求，例如get '/cats'通过网络进入。

在此示例中，我们还提供的替代（测试双精度）实现，该实现CatsService仅返回我们可以测试的硬编码值。使用overrideProvider()，以提供这样的替代实施方式。类似地，巢提供了一些方法来覆盖防护装置，拦截器，过滤器和管道与overrideGuard()，overrideInterceptor()，overrideFilter()，和overridePipe()分别的方法。

每个重写方法都会使用3种不同的方法返回一个对象，这些方法与针对自定义提供程序描述的方法类似：

  • useClass：您提供了一个将实例化的类，以提供实例来覆盖该对象（提供者，保护者等）。
  • useValue：您提供了一个将覆盖该对象的实例。
  • useFactory：您提供了一个函数，该函数返回将覆盖该对象的实例。

每种覆盖方法类型依次返回TestingModule实例，因此可以与其他方法以流畅的样式链接在一起。您应compile()在此类链的末尾使用，以使Nest实例化并初始化模块。

编译的模块具有几种有用的方法，如下表所示：

编译方法 说明
createNestApplication() INestApplication根据给定的模块创建并返回一个Nest应用程序（实例）。请注意，您必须使用init()方法手动初始化应用程序。
createNestMicroservice() INestMicroservice根据给定的模块创建并返回Nest微服务（实例）。
get() 检索应用程序上下文中可用的控制器或提供程序的静态实例（包括防护，过滤器等）。继承自模块引用类。
resolve() 检索在应用程序上下文中可用的控制器或提供程序（包括防护，过滤器等）的动态创建的作用域实例（请求或临时实例）。继承自模块引用类。
select() 浏览模块的依赖关系图；可用于从所选模块中检索特定实例（与方法中的严格模式（strict: true）一起使用get()）。

TIP

提示 将e2e测试文件保留在e2e目录中。测试文件应带有.e2e-spec或.e2e-test后缀。

# 测试请求范围的情况下

为每个传入请求唯一创建请求范围的提供程序。请求完成处理后，将垃圾回收该实例。这带来了一个问题，因为我们无法访问专门为已测试请求生成的依赖项注入子树。

我们知道（基于以上部分）该resolve()方法可用于检索动态实例化的类。此外，作为描述在这里，我们知道我们可以通过一个唯一的上下文标识符来控制DI容器子树的生命周期。我们如何在测试环境中利用这一点？

该策略是预先生成一个上下文标识符，并强制Nest使用此特定ID为所有传入请求创建一个子树。这样，我们将能够检索为已测试请求创建的实例。

为此，请jest.spyOn()在上使用ContextIdFactory：

const contextId = ContextIdFactory.create();
jest
  .spyOn(ContextIdFactory, 'getByRequest')
  .mockImplementation(() => contextId);

现在，我们可以使用contextId来访问任何后续请求的单个生成的DI容器子树。

catsService = await moduleRef.resolve(CatsService, contextId);
