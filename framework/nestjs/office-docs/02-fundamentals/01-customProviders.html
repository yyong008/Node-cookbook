# 自定义 Providers

在前面的章节中，我们介绍了依赖注入（DI）的各个方面以及如何在Nest中使用它。一个示例是基于构造函数的依赖项注入，用于将实例（通常是服务提供者）注入类。了解Dependency Injection是从根本上内置于Nest核心的，您不会感到惊讶。到目前为止，我们仅探讨了一种主要模式。随着应用程序变得越来越复杂，您可能需要利用DI系统的全部功能，因此让我们对其进行更详细的研究。

# id 基础

依赖注入是一种控制反转（IoC）技术，您可以将依赖的实例化委派给IoC容器（在我们的示例中为NestJS运行时系统），而不是必须在自己的代码中执行。让我们从“ 提供者”一章中检查此示例中发生的情况。

首先，我们定义一个提供程序。该@Injectable()装饰标记的CatsService类作为供应商。

// Cats.service.ts
import { Injectable } from '@nestjs/common';
import { Cat } from './interfaces/cat.interface';

@Injectable()
export class CatsService {
  private readonly cats: Cat[] = [];

  findAll(): Cat[] {
    return this.cats;
  }
}

然后，我们要求Nest将提供程序注入到我们的控制器类中：

// Cats.controllers.ts
import { Controller, Get } from '@nestjs/common';
import { CatsService } from './cats.service';
import { Cat } from './interfaces/cat.interface';

@Controller('cats');
export class CatsController {
  constructor(private catsService: CatsService) {}

  @Get()
  async findAll(): Promise<Cat[]> {
    return this.catsService.findAll();
  }
}

最后，我们在Nest IoC容器中注册提供程序：

import { Controller, Get, Bind, Dependencies } from '@nestjs/common';
import { CatsService } from './cats.service';

@Controller('cats');
@Dependencies(CatsService)
export class CatsController {
  constructor(catsService) {
    this.catsService = catsService;
  }

  @Get()
  async findAll() {
    return this.catsService.findAll()
  }
}

最后，我们在Nest IoC容器中注册提供程序：

import { Module } from '@nestjs/common';
import { CatsController } from './cats/cats.controller';
import { CatsService } from './cats/cats.service';

@Module({
  controllers: [CatsController],
  providers: [ CatsService ]
})
export class AppModule{}

进行这项工作的幕后到底发生了什么？此过程包含三个关键步骤：

  1. 在中cats.service.ts，@Injectable()装饰器将CatsService类声明为可由Nest IoC容器管理的类。

  2. 在中cats.controller.ts，使用构造函数注入CatsController声明对CatsService令牌的依赖关系：

constructor(private catsService: CatsService)
  1. 在app.module.ts我们令牌关联CatsService与类CatsService从cats.service.ts文件。我们将在下面确切地看到这种关联（也称为registration）是如何发生的。

当Nest IoC容器实例化一个时CatsController，它首先查找所有依赖项*。找到CatsService依赖项后，它会在注册步骤（上面的＃3）上对CatsService令牌执行查询，该令牌返回CatsService该类。假定SINGLETON范围（默认行为），Nest然后将创建的实例CatsService，对其进行缓存，然后将其返回，或者如果已被缓存，则返回现有实例。

*此说明有点简化，以说明这一点。我们忽略的一个重要方面是，分析代码的依赖关系的过程非常复杂，并且发生在应用程序引导过程中。一个关键特性是依赖性分析（或“创建依赖性图”）是可传递的。在上面的示例中，如果CatsService自身具有依赖关系，那么那些依赖关系也将得到解决。依赖关系图确保以正确的顺序解决依赖关系-本质上是“自下而上”。这种机制使开发人员不必管理此类复杂的依赖关系图。

# 标准 Providers

让我们仔细看看@Module()装饰器。在中app.module，我们声明：

@Module({
  controllers: [CatsController],
  providers: [CatsService],
})

该providers属性采用的数组providers。到目前为止，我们已经通过类名列表提供了这些提供程序。实际上，该语法providers: [CatsService]是更完整语法的简写：

providers: [
  {
    provide: CatsService,
    useClass: CatsService,
  },
];

现在，我们看到了这种显式构造，我们可以了解注册过程。在这里，我们显然将令牌CatsService与class 相关联CatsService。简写只是为了简化最常用的用例，在该用例中，令牌用于请求具有相同名称的类的实例。

# 自定义 Providers

当您的要求超出标准提供商所提供的要求时，会发生什么？这里有一些例子：

您想要创建一个自定义实例，而不是让Nest实例化一个类（或返回其缓存实例） 您想在第二个依赖项中重用现有的类 您想使用模拟版本覆盖类进行测试 Nest可让您定义自定义提供程序来处理这些情况。它提供了几种定义自定义提供程序的方法。让我们来看看它们。

# Value Providers: useValue

该useValue语法对于注入常量值，将外部库放入Nest容器或用模拟对象替换实际实现很有用。假设您想强迫Nest使用模拟CatsService进行测试。

import { CatsService } from './cats.service';

const mockCatsService = {
  /* mock implementation
  ...
  */
};

@Module({
  imports: [CatsModule],
  providers: [
    {
      provide: CatsService,
      useValue: mockCatsService,
    },
  ],
})
export class AppModule {}

在此示例中，CatsService令牌将解析为mockCatsService模拟对象。useValue需要一个值-在这种情况下，CatsService它是一个文字对象，该对象与要替换的类具有相同的接口。由于TypeScript的结构化类型，您可以使用具有兼容接口的任何对象，包括文字对象或以实例化的类实例new。

# 基于非级供应商令牌

到目前为止，我们已经使用了类名作为我们的提供程序标记（数组中provide列出的提供程序中的属性值providers）。这与基于构造函数的注入使用的标准模式匹配，其中令牌也是类名。（如果此概念尚不完全清楚，请参阅DI基础知识以重新学习令牌）。有时，我们可能希望灵活使用字符串或符号作为DI令牌。例如：


import { connection } from './connection';

@Module({
  providers: [
    {
      provide: 'CONNECTION',
      useValue: connection,
    },
  ],
})
export class AppModule {}

在此示例中，我们将字符串值标记（'CONNECTION'）与connection我们从外部文件导入的预先存在的对象相关联。

TIP

注意 除了将字符串用作标记值之外，您还可以使用JavaScript 符号

前面我们已经看到了如何使用基于标准构造函数的注入模式来注入提供程序。此模式要求使用类名声明依赖项。该'CONNECTION'定义提供使用字符串值的令牌。让我们看看如何注入这样的提供者。为此，我们使用@Inject()装饰器。这个装饰器接受一个参数-令牌。

@Injectable()
@Dependencies('CONNECTION')
export class CatsRepository {
  constructor(connection) {}
}

在此示例中，CatsService令牌将解析为mockCatsService模拟对象。useValue需要一个值-在这种情况下，CatsService它是一个文字对象，该对象与要替换的类具有相同的接口。由于TypeScript的结构化类型，您可以使用具有兼容接口的任何对象，包括文字对象或以实例化的类实例new。

# 类 Providers: useClass

到目前为止，我们已经使用了类名作为我们的提供程序标记（数组中provide列出的提供程序中的属性值providers）。这与基于构造函数的注入使用的标准模式匹配，其中令牌也是类名。（如果此概念尚不完全清楚，请参阅DI基础知识以重新学习令牌）。有时，我们可能希望灵活使用字符串或符号作为DI令牌。例如：

import { connection } from './connection';

@Module({
  providers: [
    {
      provide: 'CONNECTION',
      useValue: connection,
    },
  ],
})
export class AppModule {}

在此示例中，我们将字符串值标记（'CONNECTION'）与connection我们从外部文件导入的预先存在的对象相关联。

TIP

注意 除了将字符串用作标记值之外，您还可以使用JavaScript 符号。

前面我们已经看到了如何使用基于标准构造函数的注入模式来注入提供程序。此模式要求使用类名声明依赖项。该'CONNECTION'定义提供使用字符串值的令牌。让我们看看如何注入这样的提供者。为此，我们使用@Inject()装饰器。这个装饰器接受一个参数-令牌。

@Injectable()
export class CatsRepository {
  constructor(@Inject('CONNECTION') connection: Connection) {}
}

TIP

提示 该@Inject()装饰器从进口@nestjs/common包。

尽管我们'CONNECTION'在上面的示例中将字符串直接用于说明目的，但为了进行干净的代码组织，最佳实践是在单独的文件（例如）中定义标记constants.ts。对待它们就像对待在其自己的文件中定义并在需要的地方导入的符号或枚举一样。

# 工厂 Providers: useFactory

该useFactory语法允许动态创建提供程序。实际提供者将由工厂函数返回的值提供。工厂功能可以根据需要简单或复杂。一个简单的工厂可能不依赖任何其他提供程序。一家更复杂的工厂本身可以注入其计算结果所需的其他提供程序。对于后一种情况，工厂提供程序语法具有一对相关的机制：

工厂函数可以接受（可选）参数。 （可选）inject属性接受一组提供程序，在实例化过程中，Nest将解析这些提供程序并将其作为参数传递给工厂函数。这两个列表应该相互关联：Nest将以inject相同的顺序将列表中的实例作为参数传递给factory函数。 下面的示例演示了这一点。

const connectionFactory = {
  provide: 'CONNECTION',
  useFactory: (optionsProvider: OptionsProvider) => {
    const options = optionsProvider.get();
    return new DatabaseConnection(options);
  },
  inject: [OptionsProvider],
};

@Module({
  providers: [connectionFactory],
})
export class AppModule {}

# 别名 Providers: useExisting

该useExisting语法允许您为现有提供程序创建别名。这创建了两种访问同一提供程序的方式。在下面的示例中，（基于字符串的）令牌'AliasedLoggerService'是（基于类的）令牌的别名LoggerService。假设我们有两种不同的依赖关系，一种用于'AlilasedLoggerService'，一种用于LoggerService。如果两个依赖项都用SINGLETON范围指定，则它们都将解析为同一实例。

@Injectable()
class LoggerService {
  /* implementation details */
}

const loggerAliasProvider = {
  provide: 'AliasedLoggerService',
  useExisting: LoggerService,
};

@Module({
  providers: [LoggerService, loggerAliasProvider],
})
export class AppModule {}

# 无服务基础提供者

尽管提供商经常提供服务，但它们不限于这种用法。提供者可以提供任何价值。例如，提供者可以根据当前环境提供一系列配置对象，如下所示：

const configFactory = {
  provide: 'CONFIG',
  useFactory: () => {
    return process.env.NODE_ENV === 'development' ? devConfig : prodConfig;
  },
};

@Module({
  providers: [configFactory],
})
export class AppModule {}

# 输出 Providers

像任何提供程序一样，自定义提供程序的范围仅限于其声明模块。为了使其对其他模块可见，必须将其导出。要导出自定义提供程序，我们可以使用其令牌或完整的提供程序对象。

以下示例显示了使用令牌进行导出：

const connectionFactory = {
  provide: 'CONNECTION',
  useFactory: (optionsProvider: OptionsProvider) => {
    const options = optionsProvider.get();
    return new DatabaseConnection(options);
  },
  inject: [OptionsProvider],
};

@Module({
  providers: [connectionFactory],
  exports: ['CONNECTION'],
})
export class AppModule {}

或者，使用完整的提供程序对象导出：

const connectionFactory = {
  provide: 'CONNECTION',
  useFactory: (optionsProvider: OptionsProvider) => {
    const options = optionsProvider.get();
    return new DatabaseConnection(options);
  },
  inject: [OptionsProvider],
};

@Module({
  providers: [connectionFactory],
  exports: [connectionFactory],
})
export class AppModule {}
