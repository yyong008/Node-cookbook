# 执行上下文

Nest提供了几个实用程序类，这些类有助于简化编写可在多个应用程序上下文中运行的应用程序（例如，基于Nest HTTP Server的微服务和WebSockets应用程序上下文）。这些实用程序提供有关当前执行上下文的信息，这些信息可用于构建可在广泛的控制器，方法和执行上下文中工作的通用防护，过滤器和拦截器。

本章将介绍两个这样的类：ArgumentsHost和ExecutionContext。

# 宿主类

本ArgumentsHost类提供用于检索传递到处理程序的参数的方法。它允许选择适当的上下文（例如HTTP，RPC（微服务）或WebSocket）从中检索参数。该框架在您可能要访问它的地方提供的实例ArgumentsHost，通常称为host参数。例如，使用实例调用异常过滤器的catch()方法。ArgumentsHost

ArgumentsHost只是充当处理程序参数的抽象。例如，对于HTTP服务器应用程序（@nestjs/platform-express使用时），该host对象封装Express的[request, response, next]数组，其中request的请求对象，response响应对象以及next控制应用程序请求-响应周期的函数。另一方面，对于GraphQL应用程序，host对象包含[root, args, context, info]数组。

# 当前应用程序上下文

当构建通用的警卫，过滤器和拦截器这意味着可以跨多个应用程序上下文中运行，我们需要一种方法来确定应用程序，我们的方法是当前运行中的类型与执行此操作。getType()方法ArgumentsHost：

const type = host.getType();
if (type === 'http') {
  // HTTP application
} else if (type === 'rpc') {
  // Microservice
}

# 主机处理器参数

要检索传递给处理程序的参数数组，一种方法是使用宿主对象的getArgs()方法。

const [req, res, next] = host.getArgs();

您可以使用以下getArgByIndex()方法按索引选择特定的参数：

const request = host.getArgByIndex(0);
const response = host.getArgByIndex(1);

在这些示例中，我们按索引检索了请求和响应对象，通常不建议这样做，因为它会将应用程序耦合到特定的执行上下文。相反，通过使用host对象的一种实用程序方法切换到适合您的应用程序的应用程序上下文，可以使代码更健壮和可重用。上下文切换实用程序方法如下所示。

/**
 * Switch context to RPC.
 */
switchToRpc(): RpcArgumentsHost;
/**
 * Switch context to HTTP.
 */
switchToHttp(): HttpArgumentsHost;
/**
 * Switch context to WebSockets.
 */
switchToWs(): WsArgumentsHost;

让我们使用switchToHttp()方法重写前面的示例。该host.switchToHttp()辅助函数返回一个HttpArgumentsHost对象，它是适用于HTTP应用程序上下文。该HttpArgumentsHost对象具有两种有用的方法，可用来提取所需的对象。在这种情况下，我们还使用Express类型断言来返回本机Express类型的对象：

const ctx = host.switchToHttp();
const request = ctx.getRequest<Request>();
const response = ctx.getResponse<Response>();

同样，WsArgumentsHost并RpcArgumentsHost具有在微服务和WebSockets上下文中返回适当对象的方法。以下是用于的方法WsArgumentsHost：

export interface WsArgumentsHost {
  /**
   * Returns the data object.
   */
  getData<T>(): T;
  /**
   * Returns the client object.
   */
  getClient<T>(): T;
}

以下是用于的方法RpcArgumentsHost：

export interface RpcArgumentsHost {
  /**
   * Returns the data object.
   */
  getData<T>(): T;

  /**
   * Returns the context object.
   */
  getContext<T>(): T;
}

# 执行上下文类

ExecutionContextextend ArgumentsHost，提供有关当前执行过程的其他详细信息。像ArgumentsHost，巢提供的实例ExecutionContext中，你可能需要它的地方，比如在canActivate()一个方法后卫和intercept()一个方法拦截器。它提供以下方法

export interface ExecutionContext extends ArgumentsHost {
  /**
   * Returns the type of the controller class which the current handler belongs to.
   */
  getClass<T>(): Type<T>;
  /**
   * Returns a reference to the handler (method) that will be invoked next in the
   * request pipeline.
   */
  getHandler(): Function;
}

该getHandler()方法返回对要调用的处理程序的引用。该getClass()方法返回Controller此特定处理程序所属的类的类型。例如，在HTTP上下文中，如果当前处理的请求是一个POST请求，绑定到create()方法上CatsController，getHandler()返回到参考create()方法和getClass()返回CatsController类型（未实例）。

const methodKey = ctx.getHandler().name; // "create"
const className = ctx.getClass().name; // "CatsController"

访问对当前类和处理程序方法的引用的能力提供了极大的灵活性。最重要的是，它使我们有机会@SetMetadata()从警卫队或拦截员内部通过装饰器访问元数据集。我们在下面介绍此用例。

# 反思与元数据

Nest提供了将自定义元数据附加到@SetMetadata()装饰器中的路由处理程序的功能。然后，我们可以从班级内部访问此元数据以做出某些决定。

@Post()
@SetMetadata('roles', ['admin'])
async create(@Body() createCatDto: CreateCatDto) {
  this.catsService.create(createCatDto);
}

TIP

提示 该@SetMetadata()装饰器从导入的@nestjs/common包。

通过上面的构造，我们将roles元数据（roles是元数据键，并且['admin']是关联的值）附加到该create()方法。尽管这可行，但@SetMetadata()直接在您的路线中使用并不是一种好习惯。而是创建您自己的装饰器，如下所示：

import { SetMetadata } from '@nestjs/common';

export const Roles = (...roles: string[]) => SetMetadata('roles', roles);

这种方法更简洁，更易读，并且使用强类型。现在我们有了一个自定义的@Roles()装饰器，可以使用它来装饰该create()方法。

@Post()
@Roles('admin')
async create(@Body() createCatDto: CreateCatDto) {
  this.catsService.create(createCatDto);
}

要访问路由的角色（自定义元数据），我们将使用Reflectorhelper类，该类由框架提供，并从@nestjs/core包中公开。Reflector可以按常规方式注入到类中：

@Injectable()
export class RolesGuard {
  constructor(private reflector: Reflector) {}
}

TIP

提示 的Reflector类是从进口@nestjs/core包。

现在，要读取处理程序元数据，请使用get()方法。

const roles = this.reflector.get<string[]>('roles', context.getHandler());

该Reflector#get方法允许我们通过传入两个参数来轻松访问元数据：元数据密钥和上下文（装饰器目标），以从中检索元数据。在此示例中，指定的键为'roles'（请参考roles.decorator.ts上面的文件以及在此进行的SetMetadata()调用）。对的调用提供了上下文context.getHandler()，这导致提取当前处理的路由处理程序的元数据。请记住，getHandler()为我们提供了有关路由处理程序功能的参考。

或者，我们可以通过在控制器级别应用元数据，将其应用于控制器类中的所有路由来组织控制器。

@Roles('admin')
@Controller('cats')
export class CatsController {}

在这种情况下，要提取控制器元数据，我们context.getClass()作为第二个参数传递（以提供控制器类作为元数据提取的上下文），而不是context.getHandler()：

const roles = this.reflector.get<string[]>('roles', context.getClass());

鉴于可以在多个级别提供元数据，您可能需要从多个上下文中提取和合并元数据。在Reflector此类提供了用于帮助，这两个实用的方法。这些方法提取这两种一次控制器和控制方法的元数据，并以不同的方式将它们组合起来。

考虑以下情形，您已经'roles'在两个级别上都提供了元数据

@Roles('user')
@Controller('cats')
export class CatsController {
  @Post()
  @Roles('admin')
  async create(@Body() createCatDto: CreateCatDto) {
    this.catsService.create(createCatDto);
  }
}

如果您打算将其指定'user'为默认角色，并为某些方法选择性地覆盖它，则可能会使用该getAllAndOverride()方法。

const roles = this.reflector.getAllAndOverride<string[]>('roles', [
  context.getHandler(),
  context.getClass(),
]);

在上述create()方法的上下文中使用上述元数据运行的带有此代码的后卫将导致roles包含['admin']。

要获取两者的元数据并将其合并（此方法合并数组和对象），请使用getAllAndMerge()方法：

const roles = this.reflector.getAllAndMerge<string[]>('roles', [
  context.getHandler(),
  context.getClass(),
]);

这将导致roles包含['user', 'admin']。

对于这两种合并方法，都将元数据键作为第一个参数传递，并将元数据目标上下文数组（即，对getHandler()and和/或getClass())方法的调用）作为第二个参数传递。
