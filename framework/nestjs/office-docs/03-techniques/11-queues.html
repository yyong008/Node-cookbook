# 队列 => Queues

队列是一种强大的设计模式，可以帮助您应对常见的应用程序扩展和性能挑战。队列可以帮助您解决的一些问题示例如下：

  • 消除处理高峰。例如，如果用户可以在任意时间启动资源密集型任务，则可以将这些任务添加到队列中，而不是同步执行它们。然后，您可以使工作进程以受控方式从队列中提取任务。您可以轻松添加新的Queue使用者，以在应用程序扩展时扩展后端任务处理。

  • 分解可能会阻塞Node.js事件循环的整体任务。例如，如果用户请求需要诸如音频代码转换之类的CPU密集型工作，则可以将此任务委派给其他进程，从而释放面向用户的进程以保持响应速度。

  • 提供跨各种服务的可靠通信渠道。例如，您可以在一个流程或服务中将任务（作业）排队，而在另一流程或服务中使用它们。可以通过任何进程或服务在作业生命周期中完成，错误或其他状态更改时（通过侦听状态事件）通知您。当队列生产者或使用者失败时，它们的状态将保留，并且当节点重新启动时，任务处理可以自动重新启动。

Nest @nestjs/bull在Bull之上提供了该软件包作为抽象/包装器，Bull是流行的，受到良好支持的高性能基于Node.js的Queue系统实现。该软件包使以嵌套友好的方式轻松将Bull Queue集成到您的应用程序中。

Bull使用Redis来保留作业数据，因此您需要在系统上安装Redis。因为它是Redis支持的，所以您的Queue体系结构可以完全分布式并且与平台无关。例如，您可以在一个（或几个）节点上的Nest中运行一些Queue 生产者，消费者和侦听器，并在其他网络节点上的其他Node.js平台上运行其他生产者，消费者和侦听器。

本章介绍了@nestjs/bull软件包。我们还建议阅读Bull文档以获取更多背景信息和特定的实现细节。

# 安装 Installation

要开始使用它，我们首先安装所需的依赖项。

$ npm install --save @nestjs/bull bull
$ npm install --save-dev @types/bull

安装过程完成后，我们可以将导入BullModule到根目录中AppModule。

// app.module.ts

import { Module } from '@nestjs/common';
import { BullModule } from '@nestjs/bull';

@Module({
  imports: [
    BullModule.registerQueue({
      name: 'audio',
      redis: {
        host: 'localhost',
        port: 6379,
      },
    }),
  ],
})
export class AppModule {}

该registerQueue()方法用于实例化和/或注册队列。在使用相同凭据连接到相同基础Redis数据库的模块和进程之间共享队列。每个队列通过其name属性都是唯一的（请参见下文）。当共享队列（跨模块/进程）时，第registerQueue()一种同时运行的方法将实例化队列并为该模块注册队列。其他模块（在相同或单独的进程中）仅注册队列。队列注册会创建一个注入令牌，该令牌可用于访问给定Nest模块中的队列。

对于每个队列，传递一个包含以下属性的配置对象：

  • name: string-队列名称，将用作注入令牌（用于将队列注入到控制器/提供程序中），以及用作装饰器的参数，以将使用者类和侦听器与队列相关联。需要。
  • limiter: RateLimiter-用于控制队列作业的处理速率的选项。有关更多信息，请参见RateLimiter。可选的。
  • redis: RedisOpts-用于配置Redis连接的选项。有关更多信息，请参见RedisOpts。可选的。
  • prefix: string-所有队列键的前缀。可选的。
  • defaultJobOptions: JobOpts-用于控制新作业的默认设置的选项。有关更多信息，请参见JobOpts。可选的。
  • settings: AdvancedSettings-高级队列配置设置。这些通常不应更改。有关更多信息，请参见AdvancedSettings。可选的。

如上所述，该name属性是必需的。其余选项是可选的，可提供对队列行为的详细控制。这些直接传递给Bull Queue构造函数。在此处阅读有关这些选项的更多信息。在第二个或后续模块中注册队列时，最佳做法是忽略name配置对象中除属性以外的所有选项。仅应在实例化队列的模块中指定这些选项。

TIP

提示 通过将多个逗号分隔的配置对象传递给registerQueue()方法来创建多个队列。

由于作业保留在Redis中，因此每次实例化特定的命名队列时（例如，启动/重新启动应用程序时），它都会尝试处理以前未完成的会话中可能存在的任何旧作业。

每个队列可以具有一个或多个生产者，消费者和侦听器。使用者以特定顺序从队列中检索作业：FIFO（默认），LIFO或根据优先级。这里讨论控制队列处理顺序。

# 生产者 => Producers

作业生产者将作业添加到队列中。生产者通常是应用程序服务（嵌套提供者）。要将作业添加到队列，请首先将队列注入服务中，如下所示：

import { Injectable } from '@nestjs/common';
import { Queue } from 'bull';
import { InjectQueue } from '@nestjs/bull';

@Injectable()
export class AudioService {
  constructor(@InjectQueue('audio') private audioQueue: Queue) {}
}

TIP

提示 该@InjectQueue()装饰标识队列通过其名称，在所提供的registerQueue()方法调用（例如，'audio'）。

现在，通过调用队列的add()方法并传递用户定义的作业对象来添加作业。作业被表示为可序列化的JavaScript对象（因为这是它们在Redis数据库中的存储方式）。您通过的工作的形状是任意的；用它来表示您的工作对象的语义。

# 命名事务 => Named jobs

作业可能具有唯一的名称。这使您可以创建专门的使用者，该使用者将仅处理具有给定名称的作业。

const job = await this.audioQueue.add('transcode', {
  foo: 'bar',
});

TIP

警告 使用命名作业时，必须为添加到队列中的每个唯一名称创建处理器，否则队列将抱怨您缺少给定作业的处理器。有关使用命名作业的更多信息，请参见此处。

# 事务选项

作业可以具有与之关联的其他选项。job在Queue.add()方法中的参数之后传递选项对象。作业选项属性为：

  • priority：number-可选的优先级值。范围从1（最高优先级）到MAX_INT（最低优先级）。请注意，使用优先级会对性能产生轻微影响，因此请谨慎使用它们。
  • delay：number-等待此作业可以等待的时间（毫秒）。请注意，为了获得准确的延迟，服务器和客户端都应使其时钟同步。
  • attempts：number-尝试完成该作业之前的总尝试次数。
  • repeat：RepeatOpts-根据cron规范重复作业。参见RepeatOpts。
  • backoff：number | BackoffOpts-如果作业失败，自动重试的退避设置。请参阅BackoffOpts。
  • lifo：boolean-如果为true，则将作业添加到队列的右端而不是左侧（默认为false）。
  • timeout：number-毫秒数，作业应在此毫秒后因超时错误而失败。
  • jobId：number| string-覆盖作业ID-默认情况下，作业ID是唯一的整数，但是您可以使用此设置覆盖它。如果使用此选项，则由您来确保jobId是唯一的。如果您尝试添加具有已经存在的ID的作业，则不会添加该作业。
  • removeOnComplete：boolean | number-如果为true，则在作业成功完成时将其删除。一个数字指定要保留的作业数量。默认行为是将作业保留在完整集中。
  • removeOnFail：boolean | number-如果为true，则在所有尝试失败后将其删除。一个数字指定要保留的作业数量。默认行为是将作业保留在失败集中。
  • removeOnFail：boolean | number-如果为true，则在所有尝试失败后将其删除。一个数字指定要保留的作业数量。默认行为是将作业保留在失败集中。

以下是一些使用作业选项自定义作业的示例。

要延迟作业的开始，请使用delay配置属性。

const job = await this.audioQueue.add(
  {
    foo: 'bar',
  },
  { delay: 3000 }, // 3 seconds delayed
);

要将作业添加到队列的右端（将作业作为LIFO（后进先出）进行处理），请将lifo配置对象的属性设置为true。

const job = await this.audioQueue.add(
  {
    foo: 'bar',
  },
  { lifo: true },
);

要确定工作的优先级，请使用priority属性。

const job = await this.audioQueue.add(
  {
    foo: 'bar',
  },
  { priority: 2 },
);

# Consumers

使用者是一个类，定义用于处理添加到队列中的作业或侦听队列中的事件或两者的方法。使用@Processor()装饰器声明消费者类，如下所示：

import { Processor } from '@nestjs/bull';

@Processor('audio')
export class AudioConsumer {}

装饰器的字符串参数（例如'audio'）是要与类方法关联的队列的名称。

在使用者类中，通过用装饰@Process()器装饰处理程序方法来声明作业处理程序。

import { Processor, Process } from '@nestjs/bull';
import { Job } from 'bull';

@Processor('audio')
export class AudioConsumer {
  @Process()
  async transcode(job: Job<unknown>) {
    let progress = 0;
    for (i = 0; i < 100; i++) {
      await doSomething(job.data);
      progress += 10;
      job.progress(progress);
    }
    return {};
  }
}

transcode()每当工作者空闲并且队列中有要处理的作业时，就会调用修饰的方法（例如）。此处理程序方法将job对象作为其唯一参数。由处理程序方法返回的值存储在作业对象中，以后可以访问它，例如在完成事件的侦听器中访问。

Job对象具有多种方法，可让您与其状态进行交互。例如，上面的代码使用该progress()方法来更新作业的进度。有关完整的对象API参考，请参见此处Job。

您可以通过将作业传递给装饰器，从而指定作业处理程序方法仅处理特定类型的作业（具有特定的作业name），如下所示。您可以在给定的使用者类中有多个处理程序，分别对应于每个作业类型（）。使用命名作业时，请确保具有与每个名称相对应的处理程序。name@Process()@Process()name

@Process('transcode')
async transcode(job: Job<unknown>) { ... }

# 事件监听器 => Event listeners

当队列和/或作业状态发生更改时，Bull会生成一组有用的事件。Nest提供了一组装饰器，允许订阅一组核心的标准事件。这些是从@nestjs/bull包中导出的。

必须在使用者类中（即，在用@Processor()装饰器装饰的类中）声明事件侦听器。要监听事件，请使用下表中的装饰器之一为该事件声明一个处理程序。例如，要侦听作业进入audio队列中的活动状态时发出的事件，请使用以下构造：

import { Processor, Process } from '@nestjs/bull';
import { Job } from 'bull';

@Processor('audio')
export class AudioConsumer {

  @OnQueueActive()
  onActive(job: Job) {
    console.log(
      `Processing job ${job.id} of type ${job.name} with data ${job.data}...`,
    );
  }

由于Bull在分布式（多节点）环境中运行，因此它定义了事件局部性的概念。这个概念认识到事件可以完全在单个进程内触发，也可以在不同进程的共享队列上触发。甲本地事件是一个时的动作或状态变化触发对在本地处理的队列被产生。换句话说，当事件生产者和使用者在单个流程中是本地的时，队列中发生的所有事件都是本地的。

当跨多个进程共享队列时，我们会遇到全局事件的可能性。为了使一个进程中的侦听器接收到另一个进程触发的事件通知，它必须注册一个全局事件。

每当发出其相应事件时，就会调用事件处理程序。使用下表中显示的签名调用处理程序，从而提供对与事件相关的信息的访问。我们在下面讨论本地和全局事件处理程序签名之间的一个关键区别。

本地事件监听器 全局事件监听器 处理程序方法签名/触发时
@OnQueueError() @OnGlobalQueueError() handler(error: Error)-发生错误。error包含触发错误。
@OnQueueWaiting() @OnGlobalQueueWaiting() handler(jobId: number
@OnQueueActive() @OnGlobalQueueActive() handler(job: Job)-工作job已经开始。
@OnQueueStalled() @OnGlobalQueueStalled() handler(job: Job)-作业job已被标记为停止。这对于调试崩溃或暂停事件循环的作业工人很有用。
@OnQueueProgress() @OnGlobalQueueProgress() handler(job: Job, progress: number)-Job job的进度已更新为有价值progress。
@OnQueueCompleted() @OnGlobalQueueCompleted() handler(job: Job, result: any)结果作业job成功完成result。
@OnQueueFailed() @OnGlobalQueueFailed() handler(job: Job, err: Error)工作job因原因而失败err。
@OnQueuePaused() @OnGlobalQueuePaused() handler() 队列已暂停。
@OnQueueResumed() @OnGlobalQueueResumed() handler(job: Job) 队列已恢复。
@OnQueueCleaned() @OnGlobalQueueCleaned() handler(jobs: Job[], type: string)旧作业已从队列中清除。jobs是清理作业的数组，是清理作业type的类型。
@OnQueueDrained() @OnGlobalQueueDrained() handler() 每当队列处理完所有等待的作业时发出（即使可能存在一些尚未处理的延迟作业）。
@OnQueueRemoved() @OnGlobalQueueRemoved() handler(job: Job)作业job已成功删除。

侦听全局事件时，方法签名可能与本地签名略有不同。具体来说，任何job在本地版本中接收对象的方法签名都将在全局版本中接收jobId（number）。job在这种情况下，要获取对实际对象的引用，请使用Queue#getJob方法。应该等待此调用，因此应该声明处理程序async。例如：

@OnGlobalQueueCompleted()
async onGlobalCompleted(jobId: number, result: any) {
  const job = await this.immediateQueue.getJob(jobId);
  console.log('(Global) on completed: job ', job.id, ' -> result: ', result);
}

TIP

提示 要访问Queue对象（进行getJob()呼叫），您当然必须注入它。另外，队列必须在注入它的模块中注册。

除了特定的事件侦听器装饰器之外，您还可以将通用@OnQueueEvent()装饰器与BullQueueEvents或BullQueueGlobalEvents枚举结合使用。在此处阅读有关事件的更多信息。

# 队列管理 Queue management

队列具有一个API，可让您执行管理功能，例如暂停和恢复，检索各种状态下的作业计数等。您可以在此处找到完整队列API 。可以直接在Queue对象上调用这些方法中的任何一个，如下面的暂停/恢复示例所示。

使用pause()方法调用暂停队列。暂停的队列在恢复之前不会处理新的作业，但是当前正在处理的作业将继续直到完成。

await audioQueue.pause();

要恢复已暂停的队列，请使用resume()如下方法：

await audioQueue.resume();

# Async configuration

您可能希望异步而不是静态地传递队列选项。在这种情况下，请使用registerQueueAsync()方法，该方法提供了几种处理异步配置的方法。

一种方法是使用工厂功能：

BullModule.registerQueueAsync({
  name: 'audio',
  useFactory: () => ({
    redis: {
      host: 'localhost',
      port: 6379,
    },
  }),
});

我们的工厂的行为类似于任何其他异步提供程序（例如，可以async并且可以通过注入依赖项inject）。

BullModule.registerQueueAsync({
  name: 'audio',
  imports: [ConfigModule],
  useFactory: async (configService: ConfigService) => ({
    redis: {
      host: configService.get('QUEUE_HOST'),
      port: +configService.get('QUEUE_PORT'),
    },
  }),
  inject: [ConfigService],
});

另外，您可以使用以下useClass语法：

BullModule.registerQueueAsync({
  name: 'audio',
  useClass: BullConfigService,
});

上面的构造将BullConfigService在内部实例化，BullModule并通过调用使用它来提供一个options对象createBullOptions()。请注意，这意味着BullConfigService必须实现BullOptionsFactory接口，如下所示：

@Injectable()
class BullConfigService implements BullOptionsFactory {
  createBullOptions(): BullModuleOptions {
    return {
      redis: {
        host: 'localhost',
        port: 6379,
      },
    };
  }
}

为了防止在BullConfigService内部创建BullModule并使用从其他模块导入的提供程序，可以使用useExisting语法。

BullModule.registerQueueAsync({
  name: 'audio',
  imports: [ConfigModule],
  useExisting: ConfigService,
});

此构造的工作方式useClass与一个关键区别相同，即BullModule查找导入的模块以重用现有模块，ConfigService而不是实例化新模块。
