# 利用 TypeScript 和 GraphQL 的力量

GraphQL是一种功能强大的API查询语言，它是一个运行时，用于使用现有数据来完成这些查询。这是一种优雅的方法，可以解决REST API通常存在的许多问题。对于背景，我们建议阅读GraphQL和REST之间的比较。GraphQL与TypeScript结合使用可帮助您通过GraphQL查询开发出更好的类型安全性，从而实现端到端的键入。

在本章中，我们假设对GraphQL有基本的了解，并将重点放在如何使用内置@nestjs/graphql模块上。该GraphQLModule是围绕着一个包装阿波罗服务器。我们使用这个经过验证的GraphQL软件包来提供将GraphQL与Nest结合使用的方法。

# 安装 => Installation

首先安装所需的软件包：

npm i --save @nestjs/graphql graphql-tools graphql

根据所使用的基础平台（Express或Fastify），还必须安装apollo-server-express或apollo-server-fastify。

# 概览 => Overview

Nest提供了两种构建GraphQL应用程序的方法，即代码优先和模式优先的方法。您应该选择最适合自己的一种。GraphQL本节中的大多数章节都分为两个主要部分：一个部分，如果您首先采用代码，则应遵循；另一部分，如果您首先采用架构，则应使用。

在代码优先方法中，您使用装饰器和TypeScript类生成相应的GraphQL模式。如果您希望专门使用TypeScript并避免在语言语法之间进行上下文切换，则此方法很有用。

在模式优先方法中，真理的来源是GraphQL SDL（模式定义语言）文件。SDL是与语言无关的在不同平台之间共享架构文件的方式。Nest会基于GraphQL模式自动生成TypeScript定义（使用类或接口），以减少编写冗余样板代码的需要。

# 开始与 TypeScript 一起使用 GraphQL => Getting started with GraphQL & TypeScript

安装软件包后，我们可以导入GraphQLModule并使用forRoot()静态方法对其进行配置。

import { Module } from '@nestjs/common';
import { GraphQLModule } from '@nestjs/graphql';

@Module({
  imports: [
    GraphQLModule.forRoot({}),
  ],
})
export class AppModule {}

该forRoot()方法将选项对象作为参数。这些选项将传递到基础Apollo实例（在此处了解有关可用设置的更多信息）。例如，如果要禁用playground和关闭debug模式，请传递以下选项：

import { Module } from '@nestjs/common';
import { GraphQLModule } from '@nestjs/graphql';

@Module({
  imports: [
    GraphQLModule.forRoot({
      debug: false,
      playground: false,
    }),
  ],
})
export class AppModule {}

如前所述，这些选项将转发给ApolloServer构造函数。

# 多端点 Multiple endpoints

该@nestjs/graphql模块的另一个有用功能是能够一次服务多个端点。这使您可以决定哪个模块应包含在哪个端点中。默认情况下，GraphQL在整个应用程序中搜索解析器。要将扫描限制为仅模块子集，请使用include属性。

GraphQLModule.forRoot({
  include: [CatsModule],
}),

# 代码第一 Code first

在代码优先方法中，您使用装饰器和TypeScript类生成相应的GraphQL模式。

要使用代码优先方法，首先将autoSchemaFile属性添加到options对象：

GraphQLModule.forRoot({
  autoSchemaFile: join(process.cwd(), 'src/schema.gql'),
}),

该autoSchemaFile属性值是将在其中创建自动生成的架构的路径。或者，可以在内存中即时生成模式。要启用此功能，请将autoSchemaFile属性设置为true：

GraphQLModule.forRoot({
  autoSchemaFile: true,
}),

此处提供完整工作的代码优先示例。

# 架构第一 => Schema first

要使用架构优先方法，首先typePaths需要向选项对象添加属性。该typePaths属性指示GraphQLModule应在何处查找要编写的GraphQL SDL模式定义文件。这些文件将合并在内存中；这使您可以将架构拆分为多个文件，并将其放在解析器附近。

GraphQLModule.forRoot({
  typePaths: ['./**/*.graphql'],
}),

通常，您还需要具有与GraphQL SDL类型相对应的TypeScript定义（类和接口）。手动创建相应的TypeScript定义是多余且繁琐的。它使我们没有一个单一的真理来源-SDL中的每个更改都迫使我们也调整TypeScript定义。为了解决这个问题，@nestjs/graphql程序包可以从抽象语法树（AST）自动生成 TypeScript定义。要启用此功能，请在配置时添加options属性。definitionsGraphQLModule

GraphQLModule.forRoot({
  typePaths: ['./**/*.graphql'],
  definitions: {
    path: join(process.cwd(), 'src/graphql.ts'),
  },
}),

definitions对象的path属性指示将生成的TypeScript输出保存到何处。默认情况下，所有生成的TypeScript类型都创建为接口。要生成类，请指定outputAs值为的属性'class'。

GraphQLModule.forRoot({
  typePaths: ['./**/*.graphql'],
  definitions: {
    path: join(process.cwd(), 'src/graphql.ts'),
    outputAs: 'class',
  },
}),

每次应用程序启动时，上述方法都会动态生成TypeScript定义。或者，最好构建一个简单的脚本来按需生成这些脚本。例如，假设我们将以下脚本创建为generate-typings.ts：

import { GraphQLDefinitionsFactory } from '@nestjs/graphql';
import { join } from 'path';

const definitionsFactory = new GraphQLDefinitionsFactory();
definitionsFactory.generate({
  typePaths: ['./src/**/*.graphql'],
  path: join(process.cwd(), 'src/graphql.ts'),
  outputAs: 'class',
});

现在，您可以按需运行此脚本：

$ ts-node generate-typings

TIP

提示 您可以预先编译脚本（例如，使用tsc）并用于node执行脚本。

要为脚本启用监视模式（在任何.graphql文件更改时自动生成键入），请将watch选项传递给generate()方法。

definitionsFactory.generate({
  typePaths: ['./src/**/*.graphql'],
  path: join(process.cwd(), 'src/graphql.ts'),
  outputAs: 'class',
  watch: true,
});

此处提供完整工作模式的第一个示例。

# 异步的配置 Async configuration

当您需要异步而不是静态地传递模块选项时，请使用该forRootAsync()方法。与大多数动态模块一样，Nest提供了多种技术来处理异步配置。

一种技术是使用工厂功能：

GraphQLModule.forRootAsync({
  useFactory: () => ({
    typePaths: ['./**/*.graphql'],
  }),
}),

像其他工厂提供者一样，我们的工厂函数可以是异步的，可以通过注入依赖项inject。

GraphQLModule.forRootAsync({
  imports: [ConfigModule],
  useFactory: async (configService: ConfigService) => ({
    typePaths: configService.getString('GRAPHQL_TYPE_PATHS'),
  }),
  inject: [ConfigService],
}),

或者，您可以GraphQLModule使用类而不是工厂来配置，如下所示：

GraphQLModule.forRootAsync({
  useClass: GqlConfigService,
}),

上面的构造在GqlConfigService内部实例化GraphQLModule，使用它来创建options对象。请注意，在此示例中，GqlConfigService必须实现GqlOptionsFactory接口，如下所示。在GraphQLModule将调用createGqlOptions()所提供的类的实例化的对象上的方法。

@Injectable()
class GqlConfigService implements GqlOptionsFactory {
  createGqlOptions(): GqlModuleOptions {
    return {
      typePaths: ['./**/*.graphql'],
    };
  }
}

如果要重用现有的选项提供程序，而不是在中创建私有副本GraphQLModule，请使用useExisting语法。

GraphQLModule.forRootAsync({
  imports: [ConfigModule],
  useExisting: ConfigService,
}),
